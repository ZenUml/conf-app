Paywall Routing conf-app · Lite

Generated 2026-05-07 from a code + Mixpanel audit. Source files referenced inline.

Headline: there is no variant routing today. Every paywall impression goes to a single modal — UpgradePrompt.vue. Bystander / HeavyCreator / PersonaComparison routing existed previously and was removed; the migration completed around 2026-05-02, and the residual prompt_variant: 'legacy' property was dropped on 2026-05-07.

What triggers the modal

Two surfaces reach the same UpgradePrompt modal. They have different trigger thresholds — easy to confuse.

SurfaceFile / lineShow conditionModal opens on…Events fired on open
Header "Upgrade" badge passive src/components/Viewer/GenericViewer.vue:97 isLite && macrosCreated > 50 && !spacePaid
(NB: ignores CSS flag)		 Click on the badge paywall_triggered (ui_component=viewer_notice) → then upgrade_modal_shown from the modal mount
Pre-edit gate blocking src/forgeIndex.ts:155, src/forge-*-editor.ts isPageEditorEditBlocked(customContentId, shouldBlockActions) where shouldBlockActions = isLite && customerSuccessServiceEnabled && macrosCreated >= 100 && !spacePaid Automatic when user clicks Edit on a macro in a CSS-enrolled, ≥100-macro, unpaid Lite space paywall_blocked_edit + paywall_triggered (ui_component=viewer_notice) → then upgrade_modal_shown
Threshold mismatch is intentional. The badge appears from 50 macros on every Lite space (regardless of CSS enrollment). The editor is only blocked at 100 macros on CSS-enrolled tenants. So a 60-macro Lite space sees the badge but can still freely edit.

Key conditions, in code

src/composables/useCustomerSuccessService.ts:29-46

const shouldBlockActions = computed(() => {
  if (spacePaidStatus.value) return false       // paid space → no block, ever
  const isLite = globals.apWrapper.isLite()
  return macrosCreated.value >= MACROS_LIMIT     // 100
    && customerSuccessServiceEnabled.value       // CSS feature flag for this tenant
    && isLite
})

src/utils/paywall/preEditGate.ts

export function isPageEditorEditBlocked(customContentId, shouldBlockActions) {
  return Boolean(customContentId) && shouldBlockActions
}

Since May 3, all paywall impressions are Legacy. Any historical conversion comparison between Legacy and Bystander measures the old code, not anything that's currently on prod. If you're tempted to "borrow Bystander's tactics," that's a forward-looking design decision — not a "re-enable existing mechanic" task.

Implications for the next workstream

  • The "53% Continue editing escape" number from the earlier brainstorm broke down as 228 / 235 = ~97% on Legacy alone once you separate variants. The Legacy modal is essentially escapable in one click. That is the dominant leak.
  • Bystander's higher action-rate (~10%) was a different engagement model (notify-admin + self-identify) that no longer exists. Re-introducing it would be net-new product work, not a regression fix.
  • The badge (passive surface, 50-macro threshold, no CSS gate) and the pre-edit gate (blocking, 100-macro, CSS-gated) reach the same modal. Any modal-side change affects both surfaces. Plan accordingly.
